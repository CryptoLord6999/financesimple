# Сдача отчёта со сбросом текущего периода

## Контекст

Findr — приложение учёта крипто-обменов. Владелец сейчас формирует периодический отчёт вручную в Google Sheets. Нужна встроенная функция «сдать отчёт»: видимые на экране оборотные показатели должны сбрасываться к началу нового отчётного периода, при этом ничего не должно физически удаляться — вся история обязана оставаться доступной через фильтр периода на странице «Детали».

## Текущее поведение (`v10/index.html`)

- Модель данных: `holders{} → accounts{}`, у каждого счёта — `balance` (текущий остаток, обновляется каждой операцией) и `transactions[]` (полная история операций, никогда не удаляется кроме явного «Обнулить»).
- `calcAcc(acc, period)` считает оборот/доходы/расходы/траты/выручку/маржу, фильтруя `transactions` по периоду. Явные периоды (`today`/`week`/`month`/`custom`/`all`, через глобальный `detailPeriod`) используются деревом держателей/счетов на странице «Детали» — фильтрация по календарным датам.
- Если `period` не передан, `filterTx` возвращает ВСЕ транзакции без фильтра (поведение «за всё время»). Так вызываются:
  - `renderHome()` → `calcAcc(acc)` — показатели текущего счёта на Главной;
  - `calcAll()` — сумма `calcAcc(a)` по всем счетам всех держателей, питает сводку на странице «Отчёт» (`sumTurnover`, `sumRevenue`, `sumIncome`, `sumExpense`, `sumSpend`, `sumMargin`, `sumBalance`, `sumCapital`, `sumSite`) и верхние карточки страницы «Детали» (`detBalance`, `detCapital`, `detSite`, `detSpend`, `detRevenue`, `detMargin`) — у обоих мест сейчас это лайфтайм-цифры, фильтр периода на них не действует.
- Кнопка «Зафиксировать» (`doFixReport`) сохраняет снимок `calcAll()` + `capital` в `savedReports{key}` — это просто запись в журнал, ничего не сбрасывает.
- `capital` (глобальная переменная) и «На сайте» (`siteBalance = capital - totalBalance`) ведут себя как баланс/актив бизнеса: ручная «Корректировка капитала» + автоматическая капитализация процента с каждой операции дохода/расхода.

## Цель

Добавить «закрытие периода»: после явного действия «Сдать отчёт» оборот/доходы/расходы/выручка/траты/маржа на Главной и на «Отчёте» визуально уходят в 0 и начинают копиться заново — но:

- балансы счетов/держателей и капитал/«На сайте» никогда не сбрасываются;
- вся история операций остаётся навсегда доступной через фильтр периода на «Деталях» (сегодня/7 дней/месяц/все/свой период);
- старое поведение «просто зафиксировать» (снимок без сброса) остаётся доступным как отдельный вариант.

## Дизайн

### 1. Новое глобальное состояние: `reportPeriodStart`

ISO-дата начала текущего отчётного периода. По умолчанию не задана (`null`) — означает «с самого начала» (сохраняет сегодняшнее поведение для всех, кто ещё не разу не сдавал отчёт).

Персистентность — по аналогии с `capital`/`savedReports`:
- `saveLocal`/`loadLocal`: новый ключ localStorage `fl_rps`.
- `saveServer` (запись в Firebase) и `subscribeToData`/серверная загрузка: новое поле `reportPeriodStart`.

### 2. Изменение расчёта

В `filterTx(txs, p)` добавляется ветка: когда `p === undefined` (неявный вызов из `calcAcc(acc)` без периода), фильтровать транзакции по `t.date >= reportPeriodStart`, если `reportPeriodStart` задан; если не задан — текущее поведение (все транзакции).

Явные периоды (`'today'|'week'|'month'|'custom'|'all'`), которые использует дерево на «Деталях», не меняются — они продолжают фильтровать по календарным датам независимо от `reportPeriodStart`.

Эффект: показатели на Главной (по активному счёту) и на «Отчёте» (`calcAll()`) автоматически становятся «с момента последней сдачи отчёта» вместо «за всё время» — без изменения кода самих этих мест, только семантика дефолтной ветки `filterTx`.

### 3. Верхняя сводка «Деталей» начинает учитывать фильтр периода

Сейчас `refreshDetail()` берёт `calcAll()` (всегда лайфтайм) для всех 6 карточек, фильтр `detailPeriod` влияет только на дерево ниже.

Меняем: Траты/Выручка/Маржа в верхней сводке считаются по `detailPeriod` (как дерево ниже — через `calcAcc(a, detailPeriod)`, просуммированный по всем счетам всех держателей). Баланс/Капитал/«На сайте» остаются лайфтайм (как сейчас) — это балансовые показатели, период на них не влияет.

### 4. Кнопка «Зафиксировать» → выбор действия

`fixReportBtn.onclick` меняется с `doFixReport()` на `openReportChoice()`. `openReportChoice()` сохраняет проверку прав `canEdit()` (как сейчас в `doFixReport`) — открывать шит могут только те, кто и сегодня может зафиксировать отчёт.

Новая пара overlay/sheet (`reportChoiceOverlay`/`reportChoiceSheet`) в существующем визуальном стиле bottom-sheet (как `quickExpenseSheet`/`editBalanceSheet`). Содержимое:

- Короткое пояснение: «Сдать отчёт» сбросит на экране оборот, доходы, расходы, выручку, траты и маржу до 0 — начнётся новый отчётный период. Балансы счетов и капитал не изменятся. Полная история останется доступна в «Деталях».
- Кнопка **«Сдать отчёт»** (основная) → `doSubmitReport()`.
- Кнопка **«Просто зафиксировать»** (вторичная) → `doJustFixReport()`.

### 5. `doSubmitReport()`

Повторяет текущую логику `doFixReport` (снимок `calcAll()` в `savedReports[key]`), плюс:
- `type: 'submit'`;
- `periodStart`: значение `reportPeriodStart` ДО этой сдачи (или `null`, если его не было) — это и начало закрываемого периода для отображения диапазона дат, и значение для отмены сдачи (см. §7);
- затем `reportPeriodStart = now.toISOString()`.

### 6. `doJustFixReport()` (переименованный текущий `doFixReport`)

Логика снимка не меняется, добавляется `type: 'fix'` в сохраняемую запись. `reportPeriodStart` не трогается.

### 7. Отмена последней сдачи

В бейдже «Последний» (`lastFixedBadge`/`checkLastFixed`): если самая свежая запись `savedReports` имеет `type === 'submit'`, рядом с датой показывается действие «Отменить».

По нажатию (после `safeConfirm`) — `undoLastSubmit()`: `reportPeriodStart = entry.periodStart`, запись удаляется из `savedReports`, далее `renderAll()`/`saveLocal()`/`saveServer()`.

Если самая свежая запись имеет `type === 'fix'` (то есть после сдачи уже было что-то ещё), действие «Отменить» не показывается — защита только от немедленной ошибки, а не откат произвольно в прошлое.

`undoLastSubmit()` гейтится тем же правом `canEdit()`, что и сама сдача отчёта (а не `canDelete()`, который используется для обычного удаления записей из истории).

### 8. История отчётов и текст «Поделиться»

`renderReportsList()`: у каждой карточки появляется бейдж — зелёный «Сдан» для `type === 'submit'`, серый «Фиксация» для `type === 'fix'` и для старых записей без поля `type` (обратная совместимость).

Для записей `type === 'submit'` карточка и текст `shareReport(key)` показывают диапазон «за период {periodStart}–{date}» вместо одной даты (если `periodStart` не задан — «с начала–{date}»). Для `type === 'fix'` отображение не меняется (одна дата фиксации, как сейчас).

### 9. Видимый текущий период на странице «Отчёт»

Рядом со сводкой/кнопкой «Зафиксировать» — подпись:
- если `reportPeriodStart` задан: «Текущий период: с {дата} · {N} дней»;
- если не задан: «Текущий период: за всё время».

## Явно вне рамок

- Раздельная сдача отчёта по держателям — сдача остаётся общей для всего бизнеса (страница «Отчёт» уже называется «Сводка по всем счетам», отдельной кнопки на держателя нет).
- Любое удаление/изменение `transactions[]` — фича их не трогает вообще.
- Напоминания/уведомления о необходимости сдать отчёт.
- Сброс капитала/«На сайте» — явно исключён по решению пользователя, они ведут себя как баланс.

## Обратная совместимость

- У существующих пользователей `reportPeriodStart` не задан → поведение идентично сегодняшнему до первой сдачи отчёта.
- У существующих записей `savedReports` нет `type`/`periodStart` → отображаются как «Фиксация» с одной датой, без диапазона. Бэкфилл не требуется.
